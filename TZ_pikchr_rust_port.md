# ТЗ: полный порт Pikchr на чистый Rust (`pikchr-rs`)

> Сессия Claude Code. Цель — **pure-Rust порт** диаграммного языка Pikchr (PIC-подобный), дающий
> `pikchr(input: &str) -> Result<String /*SVG*/>`, без C и без внешних рантаймов. Артефакт
> встраивается в наш основной проект рендеринга диаграмм как отдельный `Renderer`-модуль (см. §8).

---

## 0. Контекст и зачем порт, а не FFI

Существующий крейт `pikchr` — это **FFI-обёртка над `pikchr.c`** (C-код компилируется build-скриптом).
Нам нужен **чистый Rust**: без C-зависимости, со статической линковкой без тулчейна C, безопасный по
памяти, и встраиваемый в нашу `cdylib`/`staticlib` единообразно с остальными модулями. Поэтому делаем
полноценный порт.

**Ключевой факт об исходниках (определяет всю стратегию):**

- Источник истины — **`pikchr.y`**, а НЕ `pikchr.c`.
- `pikchr.y` — грамматика для генератора парсеров **Lemon** (LALR(1), yacc-подобный). Файл на ~94%
  состоит из обычного C-кода (семантические действия, геометрия, SVG-эмиттер) и лишь на ~6% — из
  грамматических правил.
- `pikchr.c` (≈270 КБ) — **сгенерированный** Lemon-ом артефакт: бóльшая часть его объёма — таблицы
  LALR-автомата. Портировать его вручную бессмысленно и вредно.

**Вывод по стратегии порта:**

1. **6% грамматики** из `pikchr.y` → переписываем в грамматику **LALRPOP** (Rust-генератор LALR(1)
   парсеров, ближайший аналог Lemon — та же парадигма, тот же класс грамматик). Это минимизирует свой
   код парсера и сохраняет соответствие оригиналу один-к-одному.
2. **94% C-кода** (действия грамматики, вычисление геометрии, layout-логика PIC, рендер в SVG) →
   переписываем в Rust **вручную и аккуратно**, сверяясь с `pikchr.y` построчно.
3. **Lemon/lempar и сам `pikchr.c` НЕ портируем** — они не нужны: их роль берёт на себя LALPOP.

**Лицензия.** Pikchr под **Zero-Clause BSD** (0BSD, фактически public domain): можно свободно
копировать, изменять, распространять. Порт делаем под нашу лицензию + сохраняем уведомление об
авторстве D. R. Hipp в NOTICE.

## 2. Принципы

1. **Чистый Rust, без C.** Никакого FFI, никакого `cc`/build-скрипта на C. `#![forbid(unsafe_code)]`
   как цель (допустимы точечные исключения только с обоснованием).
2. **Соответствие оригиналу по поведению, а не по коду.** Эталон — наблюдаемое поведение `pikchr.y`:
   тот же входной язык, та же геометрия, эквивалентный (не обязательно побайтовый) SVG. Сверка —
   дифф-тестами против C-референса (§6).
3. **Минимум своего кода в парсере.** Грамматику отдаём LALRPOP, руками пишем только семантику.
4. **SVG-выход в нашем Qt-совместимом профиле** (см. §7) — чтобы модуль сразу годился для нашего
   проекта, а не только «как у оригинала».
5. **Изоляция.** Крейт самодостаточен, не зависит от остального нашего workspace; интеграция —
   тонкой обёрткой позже (§8).

## 3. Архитектура крейта `pikchr-rs`

Pikchr — самодостаточный конвейер `текст → SVG` со **своей** геометрической раскладкой (PIC-стиль:
курсор, относительное позиционирование, направления). Внешний layout-движок НЕ нужен и НЕ
используется. Внутреннее деление на модули:

| Модуль | Ответственность | Соответствие в `pikchr.y` |
|---|---|---|
| `lexer` | Токенизация (числа, строки, имена, единицы, операторы, комментарии) | лексер-секция `pikchr.y` |
| `grammar` (LALRPOP) | Правила грамматики → AST/команды | %-правила (те самые ~6%) |
| `ast` | Узлы: объекты, атрибуты, выражения, присваивания, макросы | C-структуры из `pikchr.y` |
| `expr` | Вычисление выражений, переменные, встроенные константы/функции | eval-код действий |
| `objects` | Типы объектов: box, circle, ellipse, line, arrow, arc, spline, dot, move, text, [...] | object-код |
| `layout` | Геометрия: курсор, направления (right/left/up/down), `.start/.end/.c/...`, относительные ссылки (`at`, `with`, `from/to`), авто-размеры, `same`, контейнеры `[]` | геометрический код |
| `render_svg` | Эмиттер SVG (наш профиль §7) | SVG-эмиттер из `pikchr.y` |
| `lib` | Публичный API `pikchr(input, flags) -> Result<String, PikchrError>` + типы ошибок | внешний `pikchr()` C-API |

### Объём языка Pikchr, который нужно покрыть (полный порт = всё это)

- **Объекты-примитивы:** `box`, `circle`, `ellipse`, `oval`, `cylinder`, `file`, `dot`, `line`,
  `arrow`, `arc`, `spline`, `move`, `text`. Плюс безымянные текстовые объекты.
- **Атрибуты:** размеры (`width/height/radius/diameter`), стиль (`color`, `fill`, `thickness`,
  `dashed`, `dotted`, `invisible`), стрелочные (`->`, `<-`, `<->`), скругление (`rad`/`chop`),
  поведение текста (`above/below/ljust/rjust/aligned/bold/italic/...`), позиционные (`at`, `with`,
  `from`, `to`, `then`, `go`, направления и расстояния).
- **Геометрия и ссылки:** именованные объекты и метки (`Foo:`), позиции (`.n/.s/.e/.w/.ne/.c/...`),
  относительные (`2nd box`, `last circle`, `previous`), векторная арифметика над позициями,
  `(x,y)`-координаты, `expr`-выражения с единицами (in/cm/px/pt/...), переменные и присваивания,
  встроенные переменные (`boxwid`, `linewid`, `scale`, ...).
- **Управляющие конструкции:** контейнеры/подблоки `[ ... ]`, `define`-макросы с параметрами,
  директивы (`direction`), `same`, `chop`.
- **Лексика:** комментарии (`#` и `/* */`), строки, числа с единицами, продолжения строк.

> Полнота = поведенческий паритет с `pikchr.y` на всём официальном `tests/`. Любую конструкцию,
> которую решено отложить, фиксировать в `UNSUPPORTED.md` с причиной (на полном порте таких быть не
> должно — отложенное допускается только как промежуточные вехи, см. §5).

## 4. Обязательные зависимости

- **Парсер:** `lalrpop` (build-time генератор) + `lalrpop-util` (runtime). LALR(1), как Lemon.
- **Лексер:** ручной, либо `logos` (быстрый derive-лексер) — выбрать в задаче P1; `logos`
  предпочтителен ради минимума своего кода, если лексика Pikchr на него ложится.
- **Числа/единицы:** стандартный `f64`; таблицу единиц переносим из `pikchr.y`.
- **SVG:** крейт `svg` (билдер) — сериализация/экранирование XML, свой XML не писать.
- **Ошибки:** `thiserror`. Опционально `miette` для позиционных сообщений (Pikchr при ошибке
  возвращает текст ошибки + исходник — это поведение воспроизвести).
- **Тесты:** `insta` (снапшоты SVG), дифф-харнесс против C-референса (§6).
- **Dev-only:** `resvg` для растеризации SVG в тестах (валидность + глазное сравнение). В рантайм НЕ
  входит.

## 5. Декомпозиция на подзадачи (вехи P0–P8)

Порт ведётся **строго инкрементально и от тестов**: каждая веха расширяет покрытие языка и сразу
закрывается дифф-тестами против C-референса. Это не параллельные субагенты — это последовательные
вехи одной сессии (между ними поведение всегда зелёное на своём подмножестве).

- **P0 — каркас и харнесс.** Создать крейт, подключить LALRPOP, сделать пустой `pikchr()` и
  **дифф-тест-харнесс** (§6): берёт `.pikchr` файл, гонит через наш порт и через C-референс,
  сравнивает геометрию/SVG. На этом этапе все тесты «ожидаемо падают» — каркас готов.
- **P1 — лексер.** Полная токенизация по `pikchr.y`. Юнит-тесты на токены, включая числа с
  единицами, строки, комментарии, продолжения строк.
- **P2 — грамматика + AST (без геометрии).** Перенести ~6% правил в LALRPOP; строить AST. Цель:
  весь официальный `tests/` **парсится без ошибок** (рендер ещё не обязателен). Снапшоты AST.
- **P3 — выражения и переменные.** `expr`-eval, единицы, присваивания, встроенные переменные/функции,
  координатная арифметика. Тесты на числовые результаты.
- **P4 — базовые объекты + базовый layout.** `box/circle/ellipse/line/arrow` + курсор, направления,
  авто-позиционирование «по умолчанию вправо», авто-размеры. Первый рабочий SVG для простых картинок;
  дифф-тест против C на подмножестве «простые цепочки объектов».
- **P5 — позиционирование и ссылки.** `at/with/from/to/then/go`, точки объекта (`.n/.c/...`),
  относительные ссылки (`last`, `2nd`, `previous`, именованные метки), `same`. Геометрия должна
  совпасть с C-референсом в пределах допуска (§6).
- **P6 — остальные объекты и атрибуты.** `arc/spline/cylinder/oval/file/dot/move/text`, все
  стрелочные/стилевые/текстовые атрибуты, скругления, `chop`. Покрыть все формы из справочника.
- **P7 — контейнеры и макросы.** Подблоки `[ ... ]` со своей системой координат, `define`-макросы с
  параметрами, `direction`. Это самые сложные для геометрии части — отдельная веха.
- **P8 — полный паритет и стабилизация.** Прогнать **весь** официальный `tests/`, добиться зелёного
  дифф-теста; обработка ошибок как у оригинала (текст ошибки + исходник); фаззинг (`cargo-fuzz`) на
  устойчивость к мусорному вводу; финальный профиль SVG §7.

**Definition of Done всего порта:** весь официальный корпус `tests/` проходит дифф-тест против
C-референса в пределах геометрического допуска; нет паник на фаззинге; `#![forbid(unsafe_code)]`
держится; крейт собирается как `lib` и встраивается обёрткой §8.

## 6. Стратегия тестирования (критично для порта)

Порт C→Rust без дифф-тестов против оригинала почти наверняка разойдётся в геометрии. Поэтому:

1. **Золотой корпус.** Собрать все `.pikchr` из официального `tests/` + добавить примеры из
   справочника языка. Это вход дифф-теста.
2. **C-референс как оракул.** Один раз собрать оригинальный `pikchr` CLI (из C) и сгенерировать
   эталонные SVG для всего корпуса. Хранить эталоны в репозитории (они стабильны). Оракул нужен
   только при обновлении эталонов, в обычном CI не собирается.
3. **Сравнение — по геометрии, не побайтово.** SVG-строки C и Rust не обязаны совпадать символ в
   символ. Сравнивать: (а) общий `viewBox`/размеры канвы в пределах допуска; (б) набор примитивов и их
   ключевые координаты (центры, концы линий, рамки) с числовым допуском eps; (в) опционально —
   растеризовать оба `resvg`-ом и сравнить пиксельно с порогом. Реализовать парсер-нормализатор SVG в
   список геометрических примитивов для устойчивого сравнения.
4. **Снапшоты (`insta`)** на AST (P2) и на наш итоговый SVG (P4+) — ловят регрессии внутри порта.
5. **Фаззинг (P8)** — `cargo-fuzz` на `pikchr()`; цель: никогда не паниковать, всегда либо SVG, либо
   аккуратная ошибка.

> Допуск eps и метрика сравнения — задаются в P0 и фиксируются; геометрия Pikchr детерминирована, так
> что допуск нужен в основном на округление float и порядок атрибутов.

## 7. Профиль SVG (для нашего проекта)

Оригинальный Pikchr эмиттит компактный SVG. Наш порт должен держаться **Qt-совместимого профиля**,
совпадающего с остальным нашим проектом:

- Разрешено: `rect`, `circle`, `ellipse`, `line`, `polyline`, `polygon`, `path`, `text`, `g`,
  presentation-атрибуты (`fill`, `stroke`, `stroke-width`, `font-*`, `text-anchor`).
- Запрещено: внешний CSS как основной механизм, `foreignObject`, сложные фильтры/маски, web-шрифты.
- Единый `viewBox`, явные `width/height`, числовые координаты.

Если оригинал использует что-то вне профиля — портируем поведение, но эмиттим в разрешённые
конструкции. Расхождения с C по способу кодирования (но не по геометрии) — допустимы и ожидаемы.

## 8. Интеграция в основной проект (после готовности порта)

`pikchr-rs` подключается как отдельный `Renderer`-модуль, **минуя общий IR** основного проекта
(Pikchr самодостаточен: свой парсер, своя раскладка, свой SVG):

```rust
// в основном проекте
pub trait Renderer {
    fn detect(&self, input: &str) -> bool;          // эвристика «это Pikchr»
    fn render(&self, input: &str) -> Result<String>; // -> SVG
}

struct PikchrRenderer;
impl Renderer for PikchrRenderer {
    fn detect(&self, s: &str) -> bool { /* напр. отсутствие @startuml и наличие pic-конструкций */ }
    fn render(&self, s: &str) -> Result<String> { pikchr_rs::pikchr(s, Default::default()) }
}
```

- Подключается под Cargo-фичей `pikchr` (по умолчанию выключена), как остальные опциональные модули.
- НЕ трогает `Diagram`/`layout-*`/общий `render-svg` — это параллельный самодостаточный путь.
- `detect`-эвристику согласовать с детектом Mermaid/PlantUML, чтобы маршрутизатор не путал нотации.

## 9. Риски и заметки

- **Геометрия — главный риск.** PIC-стиль (относительное позиционирование, направления, `chop`,
  авто-маршрутизация линий между объектами) полон краевых случаев. Митигируется дифф-тестами (§6) и
  выделением layout в отдельную, плотно покрытую тестами веху P5/P6.
- **Грамматика Lemon → LALRPOP.** Оба LALR(1), перенос правил прямой, но конфликты
  shift/reduce могут проявиться иначе — разрешать, сверяясь с поведением, а не с внутренним устройством
  Lemon.
- **`define`-макросы (P7)** — текстовая подстановка с параметрами на уровне до/во время парсинга;
  уточнить по `pikchr.y`, на каком этапе разворачиваются, и повторить тот же порядок.
- **Объём.** Это серьёзный порт (исходная семантика — десятки КБ плотного C). Реалистично — крупная
  многоэтапная работа; вехи P0–P8 сделаны так, чтобы каждая давала рабочее подмножество под тестами,
  а не «всё или ничего».
- **Сохранить авторство.** 0BSD не требует, но в NOTICE указать оригинал (D. R. Hipp, pikchr.org).
